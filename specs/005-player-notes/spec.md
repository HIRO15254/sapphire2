# Feature Specification: プレイヤーメモ機能

**Feature Branch**: `005-player-notes`
**Created**: 2026-03-28
**Status**: Implemented
**Input**: User description: "プレイヤーに任意数のタグを割り当て、エディタを使ってHTML形式でメモを作成できるようにしたい"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - プレイヤーを登録・編集・削除できる (Priority: P1)

ユーザーは `Players` 画面でプレイヤーを新規登録し、名前を編集し、不要になったプレイヤーを削除できる。作成済みプレイヤーは一覧に表示され、タグとメモの有無もあわせて確認できる。

**Why this priority**: プレイヤーはタグとメモの基盤であり、ここが使えなければ他の機能の価値が出ない。

**Independent Test**: プレイヤーを追加し、一覧に表示されること、編集で名前が更新されること、削除で一覧から消えることを確認する。

**Acceptance Scenarios**:

1. **Given** ログイン済みのユーザーが `Players` 画面を開いている, **When** 新しいプレイヤー名を入力して保存する, **Then** プレイヤーが一覧に追加される
2. **Given** 既存のプレイヤーが表示されている, **When** 編集を開いて名前を変更して保存する, **Then** 一覧の表示が更新される
3. **Given** 既存のプレイヤーが表示されている, **When** 削除を実行する, **Then** そのプレイヤーが一覧から消える

---

### User Story 2 - プレイヤーにタグを付けて管理できる (Priority: P2)

ユーザーは色付きタグを作成し、プレイヤーに割り当て、必要に応じてタグをまとめて管理できる。タグは複数プレイヤーで共有でき、一覧はタグ条件で絞り込める。

**Why this priority**: タグはプレイヤーの特徴を素早く把握するために重要で、一覧の見通しを大きく改善する。

**Independent Test**: タグを作成し、複数プレイヤーに割り当て、一覧のタグ条件で対象プレイヤーだけが表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** `Players` 画面でタグを新規作成する, **When** 名前と色を保存する, **Then** タグが一覧に追加される
2. **Given** プレイヤー編集画面でタグを選択する, **When** 保存する, **Then** そのプレイヤーのタグ表示が更新される
3. **Given** タグ管理画面でタグ名または色を変更する, **When** 保存する, **Then** そのタグを使っている表示がすべて更新される
4. **Given** タグ管理画面でタグを削除する, **When** 削除を確定する, **Then** そのタグはすべてのプレイヤーから外れる
5. **Given** 複数のタグが存在する, **When** 一覧でタグ条件を選ぶ, **Then** 選択したタグを持つプレイヤーだけが表示される

---

### User Story 3 - プレイヤーメモをリッチテキストで編集できる (Priority: P3)

ユーザーはプレイヤーごとに HTML 形式のメモを保存し、太字、見出し、リスト、リンクなどを使って読みやすい記録を残せる。保存済みメモは一覧カードで存在が分かり、開くと本文を確認できる。

**Why this priority**: メモはタグより細かい観察を残すための機能で、タグ管理よりは補助的だが、継続利用の価値が高い。

**Independent Test**: プレイヤーにメモを書いて保存し、再編集時に内容が保持され、一覧カードでメモの有無と本文が確認できることを確認する。

**Acceptance Scenarios**:

1. **Given** プレイヤー作成または編集フォームを開いている, **When** リッチテキストエディタでメモを入力して保存する, **Then** HTML 形式のメモがプレイヤーに保存される
2. **Given** 保存済みのメモがある, **When** 編集フォームを再度開く, **Then** 以前の内容がエディタに復元される
3. **Given** 保存済みのメモがある, **When** 一覧のカードを開く, **Then** メモの存在がアイコンや本文で分かる

---

### Edge Cases

- 同名のプレイヤーが登録されても区別できるか
- タグが削除されたとき、既存のプレイヤー表示が崩れないか
- メモにリンクや見出しが含まれても一覧表示で安全に扱えるか
- タグ条件を選択した状態で対象が 0 件になったとき、空状態が分かりやすいか

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは認証済みユーザーごとにプレイヤー一覧を表示しなければならない
- **FR-002**: システムはユーザーがプレイヤーを新規作成、編集、削除できるようにしなければならない
- **FR-003**: システムはプレイヤーのタグを色付きで表示しなければならない
- **FR-004**: システムはユーザーがタグを新規作成、編集、削除できるようにしなければならない
- **FR-005**: システムはユーザーがプレイヤーに複数のタグを割り当てられるようにしなければならない
- **FR-006**: システムはタグ条件でプレイヤー一覧を絞り込めなければならない
- **FR-007**: システムはプレイヤーごとに HTML 形式のメモを保存し、再編集時に復元できなければならない
- **FR-008**: システムは一覧上でメモの有無を分かるように示さなければならない
- **FR-009**: システムはタグとメモの変更を同一ユーザー内で一貫して反映しなければならない
- **FR-010**: システムはタグ色として定義済みのプリセットカラーのみを受け付けなければならない

### Key Entities

- **Player**: プレイヤー 1 件を表す。属性: id, userId, name, memo, createdAt, updatedAt, tags
- **PlayerTag**: プレイヤーに付与する共有タグを表す。属性: id, userId, name, color, createdAt, updatedAt
- **PlayerToPlayerTag**: Player と PlayerTag の関連を表す中間エンティティ。属性: playerId, playerTagId, position

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `Players` 画面でプレイヤーを作成し、一覧に即時反映される
- **SC-002**: 既存プレイヤーの名前変更と削除が一覧に反映される
- **SC-003**: タグを作成し、同じタグを複数のプレイヤーに割り当てられる
- **SC-004**: タグ条件で一覧を絞り込むと、対象のプレイヤーだけが表示される
- **SC-005**: プレイヤー編集時にメモを保存し、再表示時に内容が保持される
- **SC-006**: メモを持つプレイヤーは一覧でメモの存在が視認できる
- **SC-007**: タグ削除後も関連するプレイヤー一覧が壊れず、タグ表示が除去される

## Assumptions

- プレイヤー名の重複は許可する
- タグ色はプリセットカラーパレットから選ぶ
- 1 プレイヤーにつきメモは 1 つ
- メモはフォーム内のリッチテキストエディタで編集する
- タグ条件は一覧の絞り込み用途に限定し、検索入力は現行 UI にはない
- UI テキストは英語で統一する
