# Feature Specification: アップデートノート表示

**Feature Branch**: `009-update-notes-display`  
**Created**: 2026-04-11  
**Status**: Draft  
**Input**: User description: "アップデートノートモーダルを追加し、過去のアップデート情報を参照できるように"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 初回アクセス時のアップデートノート自動表示 (Priority: P1)

アプリケーションのアップデートがリリースされた後、ユーザーが初めてアプリにアクセスすると、最新のアップデート内容を知らせるシート（モバイルではドロワー）が自動的に開く。ユーザーは新しい機能や変更点を確認し、シートを閉じることで通常のアプリ利用を続けられる。

**Why this priority**: ユーザーが新機能や変更点を見逃さないようにするための主要機能であり、アップデートノート機能の最大の価値を提供する。

**Independent Test**: アプリのバージョンを更新した状態でログインし、シートが自動的に開くことを確認する。シートを閉じた後、再度アクセスしても自動表示されないことを確認する。

**Acceptance Scenarios**:

1. **Given** アップデートがリリースされた後、ユーザーがまだそのバージョンのアプリにアクセスしていない, **When** ユーザーがアプリにアクセスする, **Then** 最新のアップデートノートを表示するシートが自動的に開く
2. **Given** ユーザーが自動表示されたシートを閉じた, **When** 同じユーザーが再度アプリにアクセスする, **Then** シートは自動的には開かない
3. **Given** アップデートがリリースされていない（前回アクセス時と同じバージョン）, **When** ユーザーがアプリにアクセスする, **Then** シートは自動的に開かない

---

### User Story 2 - アップデートノート一覧の閲覧 (Priority: P2)

ユーザーがいつでもアップデートノートの一覧を開き、過去のすべてのアップデート情報を確認できる。一覧にはバージョン名とリリース日が表示され、まだ確認していないアップデートは視覚的に強調される。

**Why this priority**: ユーザーが過去のアップデート情報を能動的に参照できる基本機能。自動表示を見逃した場合や、過去の変更点を確認したい場合に必要。

**Independent Test**: アプリの設定やメニューからアップデートノート一覧を開き、バージョン名・リリース日が表示されることを確認する。未確認のアップデートが強調表示されていることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーがアプリを利用中, **When** アップデートノート一覧を開く操作を行う, **Then** すべてのアップデートがバージョン名とリリース日とともに一覧表示される
2. **Given** まだ一覧上で表示していないアップデートが存在する, **When** ユーザーがアップデートノート一覧を開く, **Then** 未確認のアップデートが強調表示される
3. **Given** ユーザーがアップデートノート一覧で特定のアップデートの詳細を表示した, **When** 次に一覧を開く, **Then** そのアップデートは強調表示されなくなっている

---

### User Story 3 - アップデート詳細の確認 (Priority: P3)

ユーザーがアップデートノート一覧から特定のアップデートを選択し、アコーディオンを展開して詳細な変更内容を確認できる。

**Why this priority**: アップデートの詳細情報を提供する補完的な機能。一覧表示だけでは伝えきれない具体的な変更点を確認できる。

**Independent Test**: アップデートノート一覧で任意のバージョンのアコーディオンを展開し、詳細な変更内容が表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーがアップデートノート一覧を表示している, **When** 特定のアップデートのアコーディオンを展開する, **Then** そのアップデートの詳細な変更内容が表示される
2. **Given** アコーディオンが展開されている, **When** ユーザーが再度アコーディオンをクリックする, **Then** アコーディオンが折りたたまれる
3. **Given** 未確認のアップデートのアコーディオンを展開した, **When** アコーディオンの内容が表示される, **Then** そのアップデートは「確認済み」として記録される

---

### Edge Cases

- 初めてアプリを利用するユーザー（アップデート履歴がない状態）がアクセスした場合、自動表示は行われない
- 複数のアップデートが未確認の状態で蓄積されている場合、自動表示では最新のアップデートのみが表示され、一覧では全件が未確認として強調される
- アップデートノートのデータがまだ存在しない（初期状態の）場合、一覧は空状態のメッセージを表示する
- オフライン状態でアクセスした場合、キャッシュされたデータがあればそれを表示する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、アップデートのバージョン名、リリース日、詳細な変更内容を含むアップデートノートデータを管理できなければならない
- **FR-002**: システムは、アップデートノートの一覧をバージョン名とリリース日とともに表示するモーダル/シート（デスクトップではダイアログ、モバイルではドロワー）を提供しなければならない
- **FR-003**: システムは、ユーザーごとにどのアップデートノートを確認済みかを追跡しなければならない
- **FR-004**: 未確認のアップデートノート（一覧上でまだ表示されていないバージョン）は視覚的に強調表示されなければならない
- **FR-005**: 各アップデートノートはアコーディオンUIで展開・折りたたみが可能であり、詳細な変更内容を表示できなければならない
- **FR-006**: アップデートのリリース後、ユーザーが初めてアプリにアクセスした際に、自動的にアップデートノートシートが開かなければならない
- **FR-007**: 自動表示されたシートを閉じた後、同じバージョンに対して再度自動表示は行われてはならない
- **FR-008**: ユーザーは任意のタイミングでアップデートノート一覧を手動で開くことができなければならない
- **FR-009**: アコーディオンでアップデートの詳細を表示した際、そのバージョンは「確認済み」として記録されなければならない
- **FR-010**: アップデートノートの一覧は新しいバージョンが上に来るように降順で表示されなければならない

### Key Entities

- **アップデートノート（Update Note）**: アプリの各バージョンに対応するリリース情報。バージョン名、リリース日、詳細な変更内容を持つ。リリース日の降順で表示される
- **アップデート確認状態（Update View Status）**: ユーザーとアップデートノートの関連を管理するエンティティ。各ユーザーがどのアップデートノートを確認したかを追跡する
- **ユーザー（User）**: 既存のユーザーエンティティ。アップデート確認状態と1対多の関係を持つ

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: アップデートリリース後の初回アクセス時に、ユーザーの画面にアップデート情報が自動的に表示される
- **SC-002**: ユーザーは3タップ/クリック以内でアップデートノート一覧にアクセスできる
- **SC-003**: 未確認のアップデートが視覚的に区別され、ユーザーが一目で新しい情報があることを認識できる
- **SC-004**: すべての過去のアップデート情報を一覧から参照でき、詳細を展開して確認できる
- **SC-005**: 確認済みのアップデートは次回以降の自動表示対象から除外され、ユーザーの操作を妨げない

## Assumptions

- アップデートノートのデータはアプリケーションのコードベース内に静的に定義される（管理画面からの動的な入力ではなく、リリースごとにコード内でメンテナンスされる）
- 「確認済み」の状態はユーザーがアコーディオンを展開してアップデート詳細を閲覧した時点で記録される
- デスクトップではダイアログ、モバイルではドロワーとして表示する既存のレスポンシブUIパターンを活用する
- 自動表示のトリガーは、アプリのバージョン情報とユーザーの最終確認バージョンの比較によって判定される
- アップデートノートの確認状態はサーバー側（データベース）で永続化され、異なるデバイスからのアクセスでも一貫した状態が保たれる
