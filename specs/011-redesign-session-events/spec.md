# Feature Specification: ライブセッション記録イベントの再設計

**Feature Branch**: `011-redesign-session-events`  
**Created**: 2026-04-12  
**Status**: Draft  
**Input**: User description: "ライブセッション記録のイベントを再設計する (GitHub Issue #111)"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - キャッシュゲームセッションのイベント記録 (Priority: P1)

キャッシュゲームのライブセッション中に、ユーザーはチップの追加・削除、スタック更新、オールイン記録、プレイヤーの参加・離席、メモの記録など、各種イベントを時系列で記録できる。現在の「chip_add」「stack_record」イベントを、より直感的で細分化されたイベント体系に置き換える。セッション開始時にはバイイン額、セッション終了時にはキャッシュアウト額を必ず記録する。

**Why this priority**: キャッシュゲームは最も頻繁に使用されるセッションタイプであり、イベント記録の基盤を正しく設計することが他の全機能に影響する。

**Independent Test**: キャッシュゲームセッションを開始し、全種類のイベント（チップ追加/削除、スタック更新、オールイン、プレイヤー参加/離席、メモ）を記録・編集・削除できることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーがキャッシュゲームセッションを開始する, **When** セッション開始イベントが作成される, **Then** バイイン額が必須項目として記録される
2. **Given** アクティブなキャッシュゲームセッションが存在する, **When** ユーザーがチップ追加イベントを記録する, **Then** チップ追加額がイベント履歴に記録され、収支計算に反映される
3. **Given** アクティブなキャッシュゲームセッションが存在する, **When** ユーザーがチップ削除（キャッシュアウト一部）イベントを記録する, **Then** チップ削除額がイベント履歴に記録され、収支計算に反映される
4. **Given** アクティブなキャッシュゲームセッションが存在する, **When** ユーザーがスタック更新イベントを記録する, **Then** 現在のスタック額が記録される
5. **Given** アクティブなキャッシュゲームセッションが存在する, **When** ユーザーがオールインイベントを記録する, **Then** ポットサイズ・トライアル数・エクイティ・勝利数が記録され、EV計算に利用される
6. **Given** アクティブなキャッシュゲームセッションが存在する, **When** ユーザーがメモイベントを記録する, **Then** テキストメモがイベント履歴に記録される
7. **Given** ユーザーがキャッシュゲームセッションを完了する, **When** セッション終了イベントが作成される, **Then** キャッシュアウト額が必須項目として記録される

---

### User Story 2 - トーナメントセッションのイベント記録 (Priority: P1)

トーナメントのライブセッション中に、ユーザーはスタック更新、チップ購入（リバイ・アドオン）、トーナメント情報更新、プレイヤーの参加・離席、メモの記録など、各種イベントを時系列で記録できる。現在の「tournament_stack_record」「tournament_result」イベントを、より明確に分離されたイベント体系に置き換える。セッション終了時には順位・総エントリー数・賞金を記録する（総エントリー数のみ空欄を許容）。

**Why this priority**: トーナメントはキャッシュゲームと並ぶ主要なセッションタイプであり、同時に再設計する必要がある。

**Independent Test**: トーナメントセッションを開始し、全種類のイベント（スタック更新、チップ購入、トーナメント情報更新、プレイヤー参加/離席、メモ）を記録・編集・削除できることを確認する。

**Acceptance Scenarios**:

1. **Given** アクティブなトーナメントセッションが存在する, **When** ユーザーがスタック更新イベントを記録する, **Then** 現在のスタック額が記録される
2. **Given** アクティブなトーナメントセッションが存在する, **When** ユーザーがチップ購入イベントを記録する, **Then** 購入名（リバイ・アドオン等）、コスト、チップ数が記録され、収支計算に反映される
3. **Given** アクティブなトーナメントセッションが存在する, **When** ユーザーがトーナメント情報更新イベントを記録する, **Then** 残りプレイヤー数、総エントリー数、平均スタック等の情報が更新される
4. **Given** アクティブなトーナメントセッションが存在する, **When** ユーザーがメモイベントを記録する, **Then** テキストメモがイベント履歴に記録される
5. **Given** ユーザーがトーナメントセッションを完了する, **When** セッション終了イベントが作成される, **Then** 順位・賞金が必須項目として記録され、総エントリー数は任意項目として記録される
6. **Given** 総エントリーが未確定のままトーナメントで飛んだ, **When** セッション終了イベントを作成する, **Then** 総エントリー数を空欄のまま完了できる

---

### User Story 3 - プレイヤー管理イベント（ヒーロー対応） (Priority: P2)

セッション中に同卓プレイヤーの参加・離席を記録する際、ヒーロー（自分自身）の参加・離席も同じイベントタイプで記録できる。これにより、ヒーローが途中退席・復帰するケースや、複数テーブルでの管理が容易になる。

**Why this priority**: プレイヤー管理はセッション記録の重要な付随機能であり、ヒーロー対応により記録の柔軟性が向上する。

**Independent Test**: セッション中にプレイヤー（通常プレイヤーおよびヒーロー）の参加・離席イベントを記録し、正しくテーブルプレイヤー一覧に反映されることを確認する。

**Acceptance Scenarios**:

1. **Given** アクティブなセッションが存在する, **When** ユーザーが通常プレイヤーの参加イベントを記録する, **Then** そのプレイヤーがテーブルプレイヤーとして追加される
2. **Given** アクティブなセッションが存在する, **When** ユーザーがヒーロー（自分）の参加イベントを記録する, **Then** ヒーローがテーブルプレイヤーとして追加される
3. **Given** テーブルにプレイヤーが参加している, **When** ユーザーがそのプレイヤーの離席イベントを記録する, **Then** そのプレイヤーがテーブルプレイヤーから非アクティブになる

---

### User Story 4 - セッション中断・再開 (Priority: P1)

セッション中に一時的に離席・復帰する場合や、完了済みセッションを再開する場合に、セッション中断（Session Pause）・セッション再開（Session Resume）イベントを使用する。Session Start / Session End は各セッションに1つだけ存在し、複数作成は不可とする。

キャッシュゲームの完了済みセッションを再開する場合は、既存の Session End イベント（キャッシュアウト額を含む）をスタック更新イベント（同額）＋ Session Pause イベントに分解し、現時刻に Session Resume イベントを作成することでセッションをアクティブ状態に戻す。

**Why this priority**: セッションの中断・再開はライブ記録の現実的なユースケースであり、Session Start/End の一意性制約と合わせて設計する必要がある。

**Independent Test**: セッションを中断・再開し、イベント履歴に正しく反映されることを確認する。また、完了済みキャッシュゲームセッションを再開し、キャッシュアウトがスタック更新＋中断に正しく分解されることを確認する。

**Acceptance Scenarios**:

1. **Given** アクティブなセッションが存在する, **When** ユーザーがセッション中断を記録する, **Then** Session Pause イベントがイベント履歴に追加される
2. **Given** 中断中のセッションが存在する, **When** ユーザーがセッション再開を記録する, **Then** Session Resume イベントがイベント履歴に追加され、セッションがアクティブ状態に戻る
3. **Given** 完了済みのキャッシュゲームセッション（Session End にキャッシュアウト額が記録されている）, **When** ユーザーがセッションを再開する, **Then** Session End イベントが削除され、同額のスタック更新イベント＋ Session Pause イベントが元の時刻に作成され、現時刻に Session Resume イベントが作成される
4. **Given** 1つのセッション, **When** Session Start または Session End を追加で作成しようとする, **Then** エラーとなり作成が拒否される（各セッションに1つのみ）

---

### User Story 5 - 既存データのマイグレーション (Priority: P2)

既存のセッションイベントデータが新しいイベント形式に安全に変換される。過去のデータは完全な情報保持を必須としないが、マイグレーション実行時にエラーが発生しないこと。

**Why this priority**: 既存ユーザーのデータを保護しつつ、新しいイベント体系への移行を安全に行うために必要。

**Independent Test**: 既存の全イベントタイプを含むテストデータに対してマイグレーションを実行し、エラーなく完了することを確認する。

**Acceptance Scenarios**:

1. **Given** 既存のキャッシュゲームイベント（chip_add, stack_record）が存在する, **When** マイグレーションを実行する, **Then** エラーなく新しいイベント形式に変換される
2. **Given** 既存のトーナメントイベント（tournament_stack_record, tournament_result）が存在する, **When** マイグレーションを実行する, **Then** エラーなく新しいイベント形式に変換される
3. **Given** マイグレーション後のデータ, **When** セッション詳細を表示する, **Then** マイグレーション後のイベントが正常に表示される

---

### User Story 6 - セッション完了時の収支自動計算 (Priority: P1)

新しいイベント体系に基づいて、セッション完了時の収支（P&L）が正しく自動計算される。キャッシュゲームでは Session Start のバイイン額・チップ追加/削除・Session End のキャッシュアウト額から、トーナメントではチップ購入コストと Session End の順位・賞金から収支が算出される。

**Why this priority**: 収支計算はアプリケーションの中核機能であり、イベント再設計と同時に対応が必要。

**Independent Test**: 各種イベントを記録した後にセッションを完了し、収支が正しく計算されることを確認する。

**Acceptance Scenarios**:

1. **Given** Session Start（バイイン額）・チップ追加/削除・Session End（キャッシュアウト額）が記録されたキャッシュゲームセッション, **When** セッションを完了する, **Then** 総バイイン・キャッシュアウト・損益が正しく計算される
2. **Given** チップ購入・Session End（順位・賞金）が記録されたトーナメントセッション, **When** セッションを完了する, **Then** リバイ/アドオンコスト・順位・賞金・損益が正しく計算される
3. **Given** オールインイベントが記録されたキャッシュゲームセッション, **When** セッションを完了する, **Then** EV（期待値）ベースのキャッシュアウトも正しく計算される

---

### Edge Cases

- チップ追加/削除額が0のイベントが記録された場合 → 0額は許可するが収支計算には影響しない
- セッション中にイベントの時刻を過去の別イベントより前に変更した場合 → 時刻の重複・逆転は並べ替え順序で整合性を維持する
- マイグレーション時に、旧イベントのペイロードが不正（JSONパースエラー等）な場合 → デフォルト値で変換し、エラーを発生させない
- メモイベントに空文字が入力された場合 → 空文字は許可しない（最低1文字必須）
- 既存の「tournament_result」に含まれていた順位・賞金情報の移行先 → トーナメントの Session End ペイロードに変換する
- 既存の「stack_record」に含まれていた allIns データの移行先 → スタック更新イベントとオールインイベントに分離して変換する
- 完了済みトーナメントセッションの再開 → キャッシュゲームとは異なり、トーナメント固有の再開ルールを定義する必要がある（Session End の順位・賞金データの扱い）
- Session Pause 後に Session End（完了）しようとした場合 → 中断中の状態からも直接完了できる
- 既存データで Session Start が複数存在するセッション（旧 reopen により作成）のマイグレーション → 2つ目以降の Session Start を Session Resume に変換し、その直前に Session Pause を挿入する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、キャッシュゲームセッションに対して以下のイベントタイプをサポートしなければならない: Session Start, Session End, Session Pause, Session Resume, Chips Add/Remove, Player Join, Player Leave, Update Stack, All-in, Memo
- **FR-002**: システムは、トーナメントセッションに対して以下のイベントタイプをサポートしなければならない: Session Start, Session End, Session Pause, Session Resume, Update Stack, Purchase Chips, Update Tournament Info, Player Join, Player Leave, Memo
- **FR-003**: Chips Add/Remove イベントは、チップの追加と削除の両方を1つのイベントタイプで記録できなければならない
- **FR-004**: All-in イベントは、ポットサイズ・トライアル数・エクイティ・勝利数を独立したイベントとして記録できなければならない
- **FR-005**: Purchase Chips イベントは、購入名（リバイ・アドオン等）、コスト、獲得チップ数を記録できなければならない
- **FR-006**: Update Tournament Info イベントは、残りプレイヤー数・総エントリー数・平均スタック等を記録できなければならない
- **FR-007**: Memo イベントは、ユーザーが入力した任意のテキスト（1文字以上）を記録できなければならない
- **FR-008**: Player Join / Player Leave イベントは、通常プレイヤーとヒーロー（自分自身）の両方を対象にできなければならない
- **FR-009**: Session Start / Session End イベントは各セッションに1つのみ存在し、ライフサイクルイベントとして自動生成され、手動での追加作成・削除は不可とする
- **FR-010**: キャッシュゲームの Session Start イベントはバイイン額を必須ペイロードとして記録しなければならない
- **FR-011**: キャッシュゲームの Session End イベントはキャッシュアウト額を必須ペイロードとして記録しなければならない
- **FR-012**: トーナメントの Session End イベントは順位・賞金を必須ペイロード、総エントリー数を任意ペイロードとして記録しなければならない
- **FR-013**: Session Pause イベントはセッションの一時中断を記録し、Session Resume イベントはセッションの再開を記録する
- **FR-014**: 完了済みキャッシュゲームセッションを再開する場合、Session End イベントを削除し、同時刻にスタック更新イベント（キャッシュアウト額と同額）＋ Session Pause イベントを作成し、現時刻に Session Resume イベントを作成しなければならない
- **FR-015**: 新しいイベント体系に基づいて、キャッシュゲームの収支計算（総バイイン、キャッシュアウト、損益、EV計算）が正しく動作しなければならない。バイインは Session Start から、キャッシュアウトは Session End から取得する
- **FR-016**: 新しいイベント体系に基づいて、トーナメントの収支計算（リバイ/アドオンコスト、順位、賞金、損益）が正しく動作しなければならない。順位・賞金は Session End から取得する
- **FR-017**: 既存のイベントデータを新しい形式にマイグレーションする際、エラーが発生してはならない
- **FR-018**: マイグレーション時、過去のデータは完全な情報保持を必須としないが、可能な範囲で情報を保持する
- **FR-019**: Update Stack イベントは、キャッシュゲームとトーナメントの両方で共通のイベントタイプとして使用できなければならない
- **FR-020**: 各イベントのペイロードはイベントタイプごとに定義されたスキーマに基づいてバリデーションされなければならない

### Key Entities

- **Session Event**: セッション中に発生する個別の記録。イベントタイプ、発生時刻、並び順、ペイロード（イベント固有データ）を持つ。キャッシュゲームセッションまたはトーナメントセッションのいずれか一方に紐付く。
- **Cash Game Event Types**: Session Start（バイイン額）, Session End（キャッシュアウト額）, Session Pause, Session Resume, Chips Add/Remove, Player Join, Player Leave, Update Stack, All-in, Memo の10種類
- **Tournament Event Types**: Session Start, Session End（順位・賞金・総エントリー数）, Session Pause, Session Resume, Update Stack, Purchase Chips, Update Tournament Info, Player Join, Player Leave, Memo の10種類
- **Player**: セッションテーブルに参加するプレイヤー。ヒーロー（自分自身）を含む。参加・離席イベントにより状態が管理される。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: キャッシュゲームセッションで全10種類のイベントを正常に記録・表示・編集・削除できる
- **SC-002**: トーナメントセッションで全10種類のイベントを正常に記録・表示・編集・削除できる
- **SC-003**: キャッシュゲームの Session Start にバイイン額、Session End にキャッシュアウト額が正しく記録され、収支計算に使用される
- **SC-004**: トーナメントの Session End に順位・賞金が必須で、総エントリー数が任意で記録され、収支計算に使用される
- **SC-005**: 新しいイベント体系でのキャッシュゲーム収支計算結果が、同等のデータを入力した場合に旧体系と同じ結果を返す
- **SC-006**: 新しいイベント体系でのトーナメント収支計算結果が、同等のデータを入力した場合に旧体系と同じ結果を返す
- **SC-007**: 既存の全テストデータに対してマイグレーションがエラーなく完了する
- **SC-008**: マイグレーション後のセッション詳細画面が正常に表示される
- **SC-009**: オールインイベントを独立して記録でき、EV計算が正しく動作する
- **SC-010**: メモイベントを記録・表示でき、セッションのタイムラインに正しく表示される
- **SC-011**: 完了済みキャッシュゲームセッションの再開時に、Session End がスタック更新＋Session Pause に正しく分解され、Session Resume が作成される
- **SC-012**: Session Start / Session End が各セッションに1つのみ存在することが保証される

## Assumptions

- 「Chips Add/Remove」は1つのイベントタイプで追加と削除の両方を扱う（金額の正負またはフラグで区別）
- 「All-in」イベントは現在の `stack_record` 内の `allIns` 配列を独立したイベントとして切り出したもの
- 「Update Stack」はキャッシュゲームとトーナメントで共通のイベントタイプとする（現在は `stack_record` と `tournament_stack_record` で分離）
- 「Purchase Chips」は現在の `tournament_stack_record` 内の `chipPurchases` / `chipPurchaseCounts` を独立したイベントとして切り出したもの
- 「Update Tournament Info」は現在の `tournament_stack_record` 内の `remainingPlayers` / `totalEntries` / `averageStack` 等の情報を扱うイベント。順位・賞金は Session End に移動したため含まない
- ヒーロー（Hero）の Player Join / Player Leave は、既存の `playerId` フィールドでヒーローを識別する方式を踏襲する
- Session Start は各セッションの最初のイベントとして1つだけ存在し、Session End は最後のイベントとして1つだけ存在する（現行の reopen で複数 Session Start を作成する方式は廃止）
- Session Pause / Session Resume は何度でも記録でき、セッションの中断・再開履歴を形成する
- イベントの時系列順序管理（sortOrder）の仕組みは現行を踏襲する
- キャッシュゲームの初回バイインは、現行の独立した `chip_add` イベントではなく、Session Start のペイロードに含める
- トーナメントの結果（順位・賞金）は、現行の独立した `tournament_result` イベントではなく、Session End のペイロードに含める
- トーナメントの Session End では総エントリー数のみ空欄（null）を許容する。参加人数が確定する前に飛ぶケースに対応するため
