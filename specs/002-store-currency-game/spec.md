# Feature Specification: 店舗・通貨・ゲーム設定マスターデータ管理

**Feature Branch**: `002-store-currency-game`
**Created**: 2026-03-19
**Status**: Implemented (doc sync 2026-04-08)
**Input**: User description: "アミューズメントポーカーの店舗管理、店舗独自通貨の管理（購入・ボーナス等）、開催ゲーム設定（リング・トーナメント、レート、使用通貨）のマスターデータ管理機能"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 店舗を登録・編集・削除できる

ユーザーは自分の店舗を登録し、名前やメモを後から編集できる。不要になった店舗は削除でき、紐づくゲーム設定も一緒に整理される。

**Independent Test**: `/stores` で店舗を作成し、一覧・編集・削除が完了する。

**Acceptance Scenarios**:

1. **Given** ログイン済みの状態で、**When** 店舗名を入力して登録する, **Then** 店舗一覧に追加される
2. **Given** 店舗が登録済みで、**When** 名前やメモを編集する, **Then** 変更が保存される
3. **Given** 店舗が登録済みで、**When** 店舗を削除する, **Then** 一覧から消え、関連データも整理される

---

### User Story 2 - 通貨と手動トランザクションを管理できる

ユーザーは独立した通貨を定義し、残高と取引履歴を確認できる。トランザクションは増減の記録として追加・編集・削除でき、履歴は続き読み込みできる。

**Independent Test**: `/currencies` で通貨を作成し、手動トランザクションを追加・編集・削除して、残高と履歴が更新されることを確認する。

**Acceptance Scenarios**:

1. **Given** 通貨がまだない状態で、**When** 名前と単位を入力して作成する, **Then** 通貨一覧に表示される
2. **Given** 通貨が登録済みで、**When** トランザクションを追加する, **Then** 残高と履歴に反映される
3. **Given** トランザクションが存在していて、**When** 金額・種別・日時・メモを編集する, **Then** 変更後の内容が表示される
4. **Given** 履歴が複数件ある状態で、**When** さらに読み込む, **Then** 次の履歴が追加される

---

### User Story 3 - 店舗詳細で Cash Game と Tournament を管理できる

ユーザーは店舗詳細画面で、Cash Game と Tournament をタブで切り替えて管理できる。Cash Game はブラインド、アンティ、バイイン幅、使用通貨、メモを持ち、Tournament はバイイン、エントリーフィー、スタック、バウンティ、使用通貨、メモを持つ。

**Independent Test**: `/stores/:storeId` を開き、Cash Game / Tournament タブから各設定を作成・編集できる。

**Acceptance Scenarios**:

1. **Given** 店舗詳細を開いた状態で、**When** Cash Game タブを選ぶ, **Then** Cash Game 設定一覧が見える
2. **Given** 店舗詳細を開いた状態で、**When** Tournament タブを選ぶ, **Then** Tournament 設定一覧が見える
3. **Given** Cash Game が登録済みで、**When** アーカイブする, **Then** 既定一覧から隠れる
4. **Given** アーカイブ済みの Cash Game があり、**When** 復元する, **Then** 既定一覧に戻る
5. **Given** Tournament が登録済みで、**When** アーカイブまたは復元する, **Then** 表示状態が切り替わる

---

### User Story 4 - Tournament の補助データを管理できる

ユーザーは Tournament に対して、ブラインドレベル、チップ購入、タグを別データとして管理できる。ブラインドレベルは並び替えでき、チップ購入とタグは追加・更新・削除できる。

**Independent Test**: Tournament を 1 件作成し、ブラインドレベル・チップ購入・タグを追加して順序や内容を編集できる。

**Acceptance Scenarios**:

1. **Given** Tournament が登録済みで、**When** ブラインドレベルを追加する, **Then** レベル順で表示される
2. **Given** ブラインドレベルが複数ある状態で、**When** 並び替える, **Then** 新しい順序が保存される
3. **Given** チップ購入が登録済みで、**When** 追加・編集・削除する, **Then** Tournament 本体とは別に更新される
4. **Given** タグが登録済みで、**When** 追加・削除する, **Then** Tournament のタグ一覧が更新される

---

### User Story 5 - 設定画面でトランザクション種別を管理できる

ユーザーはトランザクション種別を編集できる。種別は初回アクセス時に既定値が用意され、使用中の種別は削除できない。

**Independent Test**: `/settings` を開き、既定の種別を確認し、名前変更と削除制限を確認する。

**Acceptance Scenarios**:

1. **Given** まだ種別がない状態で、**When** 設定画面を開く, **Then** 既定の種別が作成されている
2. **Given** 種別が登録済みで、**When** 名前を変更する, **Then** 取引履歴の表示名にも反映される
3. **Given** 種別が取引で使われていて、**When** 削除する, **Then** 削除は拒否される

---

### Edge Cases

- 店舗を削除した場合、配下の Cash Game と Tournament は一緒に整理される
- 通貨を削除した場合、通貨トランザクションは削除され、ゲーム設定側の通貨参照は外れる
- 手動トランザクションは負の金額を許容する
- セッション由来のトランザクションは編集・削除できない
- アーカイブ済みゲームは既定一覧には出さない

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはユーザーごとに店舗の CRUD を提供しなければならない
- **FR-002**: 店舗には名前とメモを設定できなければならない
- **FR-003**: システムはユーザーごとに通貨の CRUD を提供しなければならない
- **FR-004**: 通貨には名前と単位表記を設定できなければならない
- **FR-005**: システムは通貨ごとの残高をトランザクション合計から算出しなければならない
- **FR-006**: システムは通貨トランザクションの CRUD と履歴の続き読み込みを提供しなければならない
- **FR-007**: システムはトランザクション種別の CRUD を提供し、既定種別 `Purchase`, `Bonus`, `Session Result`, `Other` を用意しなければならない
- **FR-008**: 使用中のトランザクション種別は削除できなければならない
- **FR-009**: システムは店舗ごとに Cash Game 設定の CRUD を提供しなければならない
- **FR-010**: Cash Game は名前、バリアント、ブラインド、アンティ、バイイン範囲、使用通貨、メモを持たなければならない
- **FR-011**: Cash Game はアーカイブと復元を提供し、既定一覧からは非表示にならなければならない
- **FR-012**: システムは店舗ごとに Tournament 設定の CRUD を提供しなければならない
- **FR-013**: Tournament は名前、バリアント、バイイン、エントリーフィー、スタック、バウンティ、使用通貨、メモを持たなければならない
- **FR-014**: Tournament はアーカイブと復元を提供し、既定一覧からは非表示にならなければならない
- **FR-015**: Tournament はブラインドレベル、チップ購入、タグを別データとして管理しなければならない
- **FR-016**: ブラインドレベルは追加・更新・削除・並び替えができなければならない
- **FR-017**: チップ購入は追加・更新・削除・並び替えができなければならない
- **FR-018**: タグは追加・削除ができなければならない
- **FR-019**: すべての操作は認証済みユーザーの所有データに限定されなければならない

### Key Entities

- **Store**: ユーザーが管理する店舗。名前とメモを持つ
- **Currency**: ユーザーが管理する通貨。名前と単位を持つ
- **TransactionType**: 通貨トランザクションの分類。既定値を持つ
- **CurrencyTransaction**: 通貨残高の増減記録。金額、種別、日時、メモを持つ
- **RingGame**: 店舗配下の Cash Game 設定。名前、ブラインド、アンティ、バイイン、通貨、メモ、アーカイブ状態を持つ
- **Tournament**: 店舗配下の Tournament 設定。名前、バイイン関連、通貨、メモ、アーカイブ状態を持つ
- **BlindLevel**: Tournament のブラインドストラクチャーの 1 行
- **TournamentChipPurchase**: Tournament のチップ購入オプション
- **TournamentTag**: Tournament に付与するタグ

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: `/stores` から店舗の登録・編集・削除ができる
- **SC-002**: `/currencies` で通貨の残高と履歴が確認できる
- **SC-003**: `/stores/:storeId` で Cash Game と Tournament をタブ切り替えで管理できる
- **SC-004**: `/settings` でトランザクション種別を編集できる
- **SC-005**: アーカイブ済みのゲームは既定一覧に表示されない
- **SC-006**: ブラインドレベルとチップ購入は並び替え後の順序で保存される

## Assumptions

- `Currencies` は独立ルートとして存在するが、現在の主要ナビには含まれていない
- `ring_game.store_id` はスキーマ上は nullable だが、作成 API と UI では店舗必須として扱う
- ゲームバリアントは現行では `nlh` のみ
- 手動トランザクションとセッション由来トランザクションは別扱いで、後者は編集・削除を禁止する
