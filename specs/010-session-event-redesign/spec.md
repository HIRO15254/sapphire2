# Feature Specification: ライブセッション記録イベントの再設計

**Feature Branch**: `010-session-event-redesign`  
**Created**: 2026-04-12  
**Status**: Draft  
**Input**: User description: "https://github.com/HIRO15254/sapphire2/issues/111"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - キャッシュゲームセッションの新イベント体系で記録する (Priority: P1)

キャッシュゲームのプレイヤーとして、セッション中に発生するイベント（チップ追加/削除、スタック更新、オールイン、メモ、プレイヤーの参加/退出）を新しいイベント体系で正確に記録したい。これにより、セッション内容をより細かく・明確に把握できるようになる。

**Why this priority**: キャッシュゲームは最も頻繁に利用されるセッションタイプであり、新イベント体系の中核となる。特にオールインが独立イベントとなることで、スタック記録と分離されてデータの明確性が向上する。

**Independent Test**: キャッシュゲームセッションを作成し、すべての新イベントタイプ（Chips Add/Remove、Update Stack、All-in、Memo、Player Join/Leave）を記録・編集・削除できることを確認する。

**Acceptance Scenarios**:

1. **Given** アクティブなキャッシュゲームセッション, **When** ユーザーがチップ追加イベントを正の金額で作成する, **Then** Chips Add イベントが記録され、セッション集計に反映される
2. **Given** アクティブなキャッシュゲームセッション, **When** ユーザーがチップ削除（負の金額）イベントを作成する, **Then** Chips Remove イベントが記録され、セッション集計に反映される
3. **Given** アクティブなキャッシュゲームセッション, **When** ユーザーがスタック更新イベントを作成する, **Then** Update Stack イベントが現在のスタック額とともに記録される
4. **Given** アクティブなキャッシュゲームセッション, **When** ユーザーがオールインイベントを作成する, **Then** All-in イベントがポットサイズ・エクイティ・トライアル数・勝利数とともに独立して記録される
5. **Given** アクティブなキャッシュゲームセッション, **When** ユーザーがメモイベントを作成する, **Then** テキストメモが時系列に沿って記録される
6. **Given** アクティブなキャッシュゲームセッション, **When** プレイヤー（ヒーロー含む）がテーブルに参加する, **Then** Player Join イベントが記録され、アクティブプレイヤーリストが更新される
7. **Given** アクティブなキャッシュゲームセッション, **When** プレイヤー（ヒーロー含む）がテーブルから退出する, **Then** Player Leave イベントが記録され、アクティブプレイヤーリストが更新される

---

### User Story 2 - トーナメントセッションの新イベント体系で記録する (Priority: P1)

トーナメントプレイヤーとして、セッション中に発生するイベント（スタック更新、チップ購入、トーナメント情報更新、メモ、プレイヤーの参加/退出）を新しいイベント体系で記録したい。これにより、リバイ・アドオンなどのチップ購入がスタック更新から分離され、トーナメント情報（残りプレイヤー数、総エントリー数）の更新も独立イベントとなる。

**Why this priority**: トーナメントはキャッシュゲームと並ぶ主要セッションタイプであり、イベント体系の再設計において同等の重要度を持つ。チップ購入とスタック更新の分離により、トーナメント固有の記録がより正確になる。

**Independent Test**: トーナメントセッションを作成し、すべての新イベントタイプ（Update Stack、Purchase Chips、Update Tournament Info、Memo、Player Join/Leave）を記録・編集・削除できることを確認する。

**Acceptance Scenarios**:

1. **Given** アクティブなトーナメントセッション, **When** ユーザーがスタック更新イベントを作成する, **Then** Update Stack イベントが現在のスタック額とともに記録される
2. **Given** アクティブなトーナメントセッション, **When** ユーザーがチップ購入イベントを作成する, **Then** Purchase Chips イベントが購入額・取得チップ数とともに記録される
3. **Given** アクティブなトーナメントセッション, **When** ユーザーがトーナメント情報を更新する, **Then** Update Tournament Info イベントが残りプレイヤー数・総エントリー数とともに記録される
4. **Given** アクティブなトーナメントセッション, **When** ユーザーがメモイベントを作成する, **Then** テキストメモが時系列に沿って記録される
5. **Given** アクティブなトーナメントセッション, **When** プレイヤー（ヒーロー含む）がテーブルに参加/退出する, **Then** Player Join/Leave イベントが記録され、プレイヤーリストが更新される

### User Story 3 - セッション集計が新イベント体系で正しく計算される (Priority: P1)

ユーザーとして、新しいイベント体系に基づいてセッションの損益計算・集計が正しく行われることを期待する。キャッシュゲームではチップ追加/削除とスタック更新から、トーナメントではチップ購入と賞金から損益が計算される。

**Why this priority**: 損益計算はアプリケーションの中核機能であり、イベント体系の変更に伴い計算ロジックが正しく動作することは必須要件である。

**Independent Test**: セッションに複数のイベントを記録した後、セッション集計（損益、プレイ時間、チップ変動）が正しく計算されることを確認する。

**Acceptance Scenarios**:

1. **Given** 複数のChips Add/Removeイベントと最終スタック記録があるキャッシュゲームセッション, **When** セッションを完了する, **Then** 損益が正しく計算される（最終スタック − 総チップイン + 総チップアウト）
2. **Given** チップ購入イベントと賞金情報があるトーナメントセッション, **When** セッションを完了する, **Then** 損益が正しく計算される（賞金 − 総チップ購入額）
3. **Given** イベントが追加・編集・削除されたセッション, **When** イベント変更が行われるたび, **Then** セッション集計が自動的に再計算される

---

### User Story 4 - 既存データが新イベント体系へ移行される (Priority: P2)

既存ユーザーとして、過去のセッションデータが新しいイベント体系に移行され、アプリケーションが引き続き正常に動作することを期待する。完全なデータ保持は必須ではないが、移行時にエラーが発生しないこと。

**Why this priority**: 既存ユーザーの体験を壊さないために必要だが、新機能の実装が優先される。

**Independent Test**: 既存データが含まれるデータベースに対してマイグレーションを実行し、エラーなく完了すること、移行後のセッション表示が正常であることを確認する。

**Acceptance Scenarios**:

1. **Given** 旧イベント形式のキャッシュゲームデータ, **When** マイグレーションを実行する, **Then** エラーなく完了し、セッションが表示可能である
2. **Given** 旧イベント形式のトーナメントデータ, **When** マイグレーションを実行する, **Then** エラーなく完了し、セッションが表示可能である
3. **Given** 旧形式の stack_record に埋め込まれたオールインデータ, **When** マイグレーションを実行する, **Then** データが新形式に変換される（可能な範囲で）

---

### User Story 5 - イベントの表示と操作UI (Priority: P2)

ユーザーとして、セッション内のイベントタイムラインで新しいイベントタイプが適切に表示され、作成・編集・削除の操作ができることを期待する。

**Why this priority**: ユーザーが新イベント体系を利用するためにUIの対応が不可欠だが、バックエンドのイベント定義・集計ロジックが先行する。

**Independent Test**: セッション画面でイベントタイムラインを開き、各イベントタイプの作成・表示・編集・削除が正しく動作することを確認する。

**Acceptance Scenarios**:

1. **Given** セッション画面, **When** ユーザーがイベント追加ボタンを押す, **Then** セッションタイプに応じた新イベントタイプ一覧が表示される
2. **Given** 各イベントタイプ, **When** ユーザーがイベントを作成する, **Then** 適切な入力フォームが表示され、データを入力できる
3. **Given** 記録済みイベント, **When** ユーザーがイベントを編集する, **Then** イベント固有の編集フォームが表示される
4. **Given** 記録済みのライフサイクル以外のイベント, **When** ユーザーが削除を試みる, **Then** 確認ダイアログ後にイベントが削除される
5. **Given** Memoイベント, **When** タイムラインに表示される, **Then** メモのテキスト内容がインラインで確認できる

---

### Edge Cases

- 既存の `stack_record` イベントにオールインデータが埋め込まれている場合、マイグレーション時にオールインを独立イベントとして抽出するか、既存ペイロード内に残すかの判断が必要
- チップ削除（Remove）で現在のチップ総額が負になる場合のバリデーション
- トーナメントの Purchase Chips で金額が0またはチップ数が0の場合の扱い
- Player Join でヒーロー（セッションオーナー）が退出後に再度参加するケース（休憩からの復帰）
- 旧形式の `tournament_result` イベントが新イベント体系でどう扱われるか（セッション完了時の処理に統合される想定）
- Memoイベントの文字数制限

## Requirements *(mandatory)*

### Functional Requirements

**キャッシュゲームイベント**

- **FR-001**: キャッシュゲームセッションでは以下のイベントタイプを手動作成できること: Chips Add (Remove)、Update Stack、All-in、Memo、Player Join、Player Leave
- **FR-002**: Session Start と Session End はシステムが自動生成するイベントであり、手動作成はできないこと（既存動作と同様）
- **FR-003**: Chips Add (Remove) イベントは正の金額（追加）および負の金額（削除）を記録できること
- **FR-004**: All-in イベントはポットサイズ、エクイティ（0〜100）、トライアル数、勝利数を独立して記録できること（従来のスタック記録内の埋め込みから分離）
- **FR-005**: Memo イベントはテキストメモを記録できること

**トーナメントイベント**

- **FR-006**: トーナメントセッションでは以下のイベントタイプを手動作成できること: Update Stack、Purchase Chips、Update Tournament Info、Memo、Player Join、Player Leave
- **FR-007**: Purchase Chips イベントは購入コストと取得チップ数を記録できること（従来の tournament_stack_record 内の埋め込みから分離）
- **FR-008**: Update Tournament Info イベントは残りプレイヤー数および総エントリー数を記録できること（従来の tournament_stack_record 内の埋め込みから分離）

**共通**

- **FR-009**: Player Join / Player Leave イベントはヒーロー（セッションオーナー）自身の参加・退出も記録できること
- **FR-010**: Update Stack イベントはキャッシュゲーム・トーナメント両方で共通のイベントタイプとし、現在のスタック額を記録すること
- **FR-011**: すべてのイベントは時系列順に記録され、発生時刻の編集ができること
- **FR-012**: ライフサイクルイベント（Session Start / Session End）以外のイベントは削除できること
- **FR-013**: イベントの作成・編集・削除のたびにセッション集計が自動再計算されること

**マイグレーション**

- **FR-014**: 旧イベント形式から新イベント形式へのデータ移行時にエラーが発生しないこと
- **FR-015**: 旧形式のデータは完全な保持を必須としないが、移行後もセッションの表示・操作が正常に行えること

### Key Entities

- **Session Event**: セッション内で発生した個々の記録。イベントタイプ、発生時刻、ソート順、ペイロード（イベント固有データ）を持つ。キャッシュゲームセッションまたはトーナメントセッションに紐づく
- **Cash Game Event Types**: Session Start、Session End、Chips Add (Remove)、Update Stack、All-in、Memo、Player Join、Player Leave
- **Tournament Event Types**: Session Start、Session End、Update Stack、Purchase Chips、Update Tournament Info、Memo、Player Join、Player Leave
- **Session Table Player**: テーブルに参加しているプレイヤー情報。Player Join/Leave イベントにより状態が管理される。ヒーロー（セッションオーナー）も同様に管理可能

## Assumptions

- イベント名称は一貫性があれば変更可能（Issue記載の名称は仮のもの）
- 旧形式の `tournament_result` イベントの情報（順位、賞金など）はセッション完了時の処理に統合され、独立イベントとしては新体系に含まれない
- 旧 `stack_record` のオールインデータは、マイグレーション時に可能な範囲で独立イベントに変換する
- 旧 `tournament_stack_record` のチップ購入データ・トーナメント情報は、マイグレーション時に可能な範囲で独立イベントに分離する
- Memo イベントのテキストには実用的な文字数上限を設ける（例: 500文字）
- 共通の Update Stack はキャッシュゲームとトーナメントで同じイベントタイプを使用する

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: キャッシュゲームセッションで8種類（Session Start、Session End、Chips Add/Remove、Update Stack、All-in、Memo、Player Join、Player Leave）すべてのイベントタイプが記録・表示できる
- **SC-002**: トーナメントセッションで8種類（Session Start、Session End、Update Stack、Purchase Chips、Update Tournament Info、Memo、Player Join、Player Leave）すべてのイベントタイプが記録・表示できる
- **SC-003**: 新イベント体系でのセッション損益計算が、旧体系と同等の精度で行われる
- **SC-004**: 既存データに対するマイグレーションがエラーなく完了する
- **SC-005**: ユーザーが新しいイベントタイプ（Memo、All-in、Purchase Chips、Update Tournament Info）を初回で迷わず使用できる
- **SC-006**: イベント作成から集計反映までの応答がユーザー体感上即座に行われる
